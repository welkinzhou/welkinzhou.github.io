### State 管理

这里引入了 `LoopState` 的概念。原始的代码中，所有循环都在一个主线程跑，默认用户很有耐心，程序不会崩溃。加入程序执行出现问题，应该能恢复到之前的状态。用户有了新的想法，应该也能调整实现方向。这个问题，在之前测试中，已经出现了好几遍。一个复杂的任务，只能看着对话继续。或者强行退出程序，调整问题重新开始，哪怕前面几步是相同步骤。状态管理并不能完全解决这个问题，但状态是我们解决这个问题必须要有的东西。基础 state 模型如下：

```python
@dataclass
class LoopState:
    messages: list  # 历史消息
    turn_count: int = 1  # 循环次数
    transition_reason: str | None = None  # 继续循环原因
```

`transition_reason` 代表继续循环的原因，例如，模型判断对话需要继续，或者异常退出、用户调整输入，都是我们可以拓展的类型。`messages` 并不止是对话消息，程序异常时，我们也可以插入自己需要的 message，例如，程序崩溃。这部分消息无需传递给模型，程序内部需要维护了解情况。有了 transition_reason 和 messages，我们就能知道当前轮是什么情况，接下来要怎么处理。

### Tool context

我之前添加的 ToolRegistry，已经完成了 Tool context 的部份功能。基础实现如下：

```python
class ToolUseContext:
    def __init__(self):
        self.handlers = {}
        self.permission_context = {}
        self.mcp_clients = {}
        self.messages = []
        self.app_state = {}
        self.notifications = []
```

除了工具集的隔离，还有运行环境隔离。比如子 agent 创建的任务，只能操作特定文件夹，需要隔离环境，阻塞任务，完成后需要发通知。
程序运行时，根据不同的环境要求，创建自己的执行环境。调用工具时，根据路由，自动找到执行环境运行工具。

```python
def route_tool(tool_name: str, tool_input: dict, ctx: ToolUseContext):
    if tool_name.startswith("mcp__"):
        # 找到 mcp 工具执行环境
        return run_mcp_tool(tool_name, tool_input, ctx)
    return run_native_tool(tool_name, tool_input, ctx)
```

除了环境隔离外，还有 tool 的调度控制。例如，read_file 这种命令，可以同时运行多个，只读不会影响结果。在需要读取某个目录结构时，并发读取更合适。涉及到文件修改的，就不能并发执行了。多个工具同时修改一个文件，很容易出现混乱。

```python
def is_concurrency_safe(tool_name: str) -> bool:
    """检查工具是否并发安全"""
    # 并发安全工具列表
    concurrency_safe_tools = ["read_file", "search_files"]
    return tool_name in concurrency_safe_tools
```

确定是否并非安全，接着分批执行，可能并发的并发执行，不能并发的串行执行

```python
batches = partition_tool_calls(tool_uses)

for batch in batches:
    if batch["is_concurrency_safe"]:
        run_concurrently(batch["blocks"])
    else:
        run_serially(batch["blocks"])

```
