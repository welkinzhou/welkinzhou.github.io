#### Agent 模式

Anthporic 上一篇博客，[Building effective agents](https://www.anthropic.com/engineering/building-effective-agents)，提到了两种模式，**Workflow** 和 **Agent**。我感觉开发的第一反应总会是 workflow，workflow 就像写代码，自己心里想好每步做什么，按顺序执行。和写代码有区别，但是不大。好处就是可控，整体流程还是在开发手中控制。Agent 模式就跟灵活了，workflow 只是在关键节点运用 LLM 的推理能力。既然有推理能力，干嘛不用呢，为什么要限制？直接让 LLM 全盘接管不就行了吗。

举个简单的例子，对于电商来说，可能有咨询下单的客服、负责售后的客服。Workflow 模式下，我们可能会指定，下单去调下单的 LLM call，售后调用售后的。每一个流程有自己特定的工具，专注于当前功能。

不过既然 LLM 有推理能力，直接让它决定下一步干什么不就行了嘛。Agent 模式就是这样的。我们只需要给出指令，LLM 根据它获取的资源，确定执行的方案，操作完成，等待反馈，修改方案，继续执行直到任务结束。这种模式就很适合做一些通用型的功能，一开始并不清楚自己具体想要实现什么，通过输入和 LLM 的反馈，慢慢确定方向。同样的，这也意味着，会牺牲一些效率，要进行多轮对话，多次修改。

这里需要明白一点，AI 当今最大的能力就是推理能力，除此之外，并没有什么新东西，它不能无中生有。这一点很重要。使用 AI 的过程，对我们自己也是一种考验，不能以其昏昏使人昭昭。例如，你只是告诉 AI 我想吃饭，你并不知道自己想吃什么，AI 告诉你的大概还不如自己去美食节走一走。决定 AI 反馈的，是你给出的描述（prompt），越详细，反馈越好，tools 也是描述的一环。Tools 作用，是帮助 AI 自己补全你没有提供的内容，例如搜索已有的文件。

### Coding

概念总是模糊不清的，代码就很清爽。learn-claude-code 的介绍，看来很有雄心，也可以说是激进。总结就是信任大模型，理解我们要做的事情，专注在我们自己的领域上，让一切变得透明简单。对于大多数人，开发 Agent 就是帮助大模型工作。模型能拿到的多少信息，意味着工作有多精确，所以工作重心就是帮助模型获取信息。总所周知，模型是有上下文窗口限制的。这也意味着，我们能传递的信息是有上限的，舍弃无用的信息，强调重要的信息，提供必要的补充内容，就要做 tradeoff。

第一课，轻松一点，体验一下 LLM 的魔法。

核心代码，一个死循环，反复执行对话和工具调用：

```python
def agent_loop(query):
  	# 用户输入
    messages = [{"role": "user", "content": query}]
    while True:
      	# 调用大模型获取结果
        response = client.messages.create(
            model=MODEL, system=SYSTEM, messages=messages,
            tools=TOOLS, max_tokens=8000,
        )
        # 将返回内容追加进消息队列
        messages.append({"role": "assistant", "content": response.content})

        if response.stop_reason != "tool_use":
            return

        results = []
        # 调用 tools
        for block in response.content:
            if block.type == "tool_use":
                output = run_bash(block.input["command"])
                results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": output,
                })
        # 收集 tools 返回信息
        messages.append({"role": "user", "content": results})
```

可以看到就是一个简单循环，核心代码看完了，补充一下其余的实现。项目用的 anthropic 调模型，我使用的是 deepseek 做测试。实例化 Anthropic 需要传入 base_url 和 api_key。deepseek 也支持 anthropic 类型的接口调用，请求地址是 `https://api.deepseek.com/anthropic`。

主要是 tools 使用，tools 用来指定 agent 可以使用的工具，每个工具可用四个字段，参考[这里](https://platform.claude.com/docs/en/agents-and-tools/tool-use/implement-tool-use)：

| Parameter        | Description                                                                                                                                                                                                                                  |
| :--------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `name`           | The name of the tool. Must match the regex `^[a-zA-Z0-9_-]{1,64}$`.                                                                                                                                                                          |
| `description`    | A detailed plaintext description of what the tool does, when it should be used, and how it behaves.                                                                                                                                          |
| `input_schema`   | A [JSON Schema](https://json-schema.org/) object defining the expected parameters for the tool.                                                                                                                                              |
| `input_examples` | (Optional) An array of example input objects to help Claude understand how to use the tool. See [Providing tool use examples](https://platform.claude.com/docs/en/agents-and-tools/tool-use/implement-tool-use#providing-tool-use-examples). |

注意这里的 tools，实现和描述是分开的。模型了解工具功能，依赖的是 description，而不是读你的源码。本来描述和实现就是分开，调用 API 很难犯描述和实现不统一的情况。LangChain 中有 tool 装饰器，可以快捷生成 tool，里面的功能描述依赖于方法注释，这种情况就会有注释和实现不匹配的情况，导致 bug。

代码中只有一个 bash tool，先看源代码：

```python
TOOLS = [{
    "name": "bash",
    "description": "Run a shell command.",
    "input_schema": {
        "type": "object",
        "properties": {"command": {"type": "string"}},
        "required": ["command"],
    },
}]


def run_bash(command: str) -> str:
    dangerous = ["rm -rf /", "sudo", "shutdown", "reboot", "> /dev/"]
    if any(d in command for d in dangerous):
        return "Error: Dangerous command blocked"
    try:
        r = subprocess.run(command, shell=True, cwd=os.getcwd(),
                           capture_output=True, text=True, timeout=120)
        out = (r.stdout + r.stderr).strip()
        return out[:50000] if out else "(no output)"
    except subprocess.TimeoutExpired:
        return "Error: Timeout (120s)"
```

这里就能感受到，实现和描述的分离，其实你不实现 run_bash，每次调命令直接添加 message，也能玩。大模型有一个大问题，就是推理会迎合客户输入，这也是幻觉常常出现的原因。比如说，给了一个加法 tool，你告诉模型调用结果显示 `1 + 1 = 3`，大模型就会顺应这个结果进行推理，哪怕这不合理。所以，tools 开发至关重要。

这里我为了方便后续拓展，做了一些封装：

```python
@dataclass(frozen=True, slots=True)
class ToolSpec:
    """A JSON-schema tool spec compatible with Anthropic tool calling."""

    name: str
    description: str
    input_schema: JsonObject

    def to_anthropic(self) -> JsonObject:
        # 将工具规范转换为 Anthropic 格式
        return {
            "name": self.name,
            "description": self.description,
            "input_schema": self.input_schema,
        }
class Tool(Protocol):
    spec: ToolSpec
		# tool 需要有 run
    def run(self, tool_input: JsonObject) -> str: ...

# 工具集，后续拓展工具都从工具集中获取
@dataclass(frozen=True, slots=True)
class ToolRegistry:
    tools: tuple[Tool, ...]

    # 注册工具
    @classmethod
    def from_tools(cls, tools: Iterable[Tool]) -> "ToolRegistry":
        return cls(tuple(tools))

    def tool_specs(self) -> list[JsonObject]:
        # 将工具集规范转换为 Anthropic 格式
        return [t.spec.to_anthropic() for t in self.tools]

    def get(self, name: str) -> Tool | None:
        for t in self.tools:
            if t.spec.name == name:
                return t
        return None
# bash 工具
@dataclass(frozen=True, slots=True)
class BashTool:
    # 超时时间，默认120秒
    timeout_s: int = 120
    # 最大输出字符数，默认50000字符
    max_output_chars: int = 50_000

    @property
    def spec(self) -> ToolSpec:
        # 返回工具规范
        return ToolSpec(
            name="bash",
            description="Run a shell command.",
            input_schema={
                "type": "object",
                "properties": {"command": {"type": "string"}},
                "required": ["command"],
            },
        )

    def run(self, tool_input: JsonObject) -> str:
        # 获取命令
        command = tool_input.get("command")
        if not isinstance(command, str):
            return "Error: Invalid input; expected {'command': string}"
        # 危险命令
        dangerous = ["rm -rf /", "sudo", "shutdown", "reboot", "> /dev/"]
        if any(d in command for d in dangerous):
            return "Error: Dangerous command blocked"

        try:
            # 执行命令
            r = subprocess.run(
                command,
                shell=True,
                cwd=os.getcwd(),
                capture_output=True,
                text=True,
                timeout=self.timeout_s,
            )
            # 获取输出，标准输出加错误输出
            out = (r.stdout + r.stderr).strip()
            # 返回输出，如果输出为空，返回 "(no output)"
            # 截取最大输出字符数
            return out[: self.max_output_chars] if out else "(no output)"
        # 超时错误
        except subprocess.TimeoutExpired:
            # 返回超时错误
            return f"Error: Timeout ({self.timeout_s}s)"

# tools 执行
@dataclass(frozen=True, slots=True)
class ToolRunner:
    registry: ToolRegistry
    on_tool_use: Callable[[str, JsonObject], None] | None = None

    def run_from_response_content(self, response_content: Iterable[Any]) -> list[JsonObject]:
        """Execute tool_use blocks and return Anthropic tool_result blocks."""
        results: list[JsonObject] = []
        for block in response_content:
            # 如果不是工具使用块，则跳过
            if getattr(block, "type", None) != "tool_use":
                continue
            # 获取工具名称
            tool_name = getattr(block, "name", None)
            # 获取工具输入
            tool_input = getattr(block, "input", None)
            # 获取工具 ID
            # tool_use 一定会返回 id
            # Anthropic 通过这个 id 将工具结果与工具使用关联起来
            tool_id = getattr(block, "id", None)

            if not isinstance(tool_name, str) or not isinstance(tool_input, dict) or not isinstance(tool_id, str):
                continue

            tool = self.registry.get(tool_name)
            if tool is None:
                output = f"Error: Unknown tool '{tool_name}'"
            else:
                if self.on_tool_use is not None:
                    self.on_tool_use(tool_name, tool_input)
                output = tool.run(tool_input)
            # 将工具结果添加到结果列表
            # tool_use_id 告诉 Anthropic 这个结果是哪个工具使用的
            results.append({"type": "tool_result", "tool_use_id": tool_id, "content": output})
        return results


```

调整后的 agent_loop 如下：

```python
def agent_loop(
    *,
    messages: list[JsonObject],
    client: Any,
    tools: list[JsonObject],
    tool_runner: ToolRunner,
    config: AgentLoopConfig,
) -> Any:
    """Run the core tool-use loop until the model stops calling tools."""

    while True:
        # 请求大模型，返回响应
        response = client.messages.create(
            model=config.model,
            system=config.system,
            messages=messages,
            tools=tools,
            max_tokens=config.max_tokens,
        )

        print(f"response.content: {response.content}")

        # 将响应添加到消息列表
        messages.append({"role": "assistant", "content": response.content})

        # 如果模型没有调用工具，则返回响应
        if response.stop_reason != "tool_use":
            return response
        # 执行工具，收集结果
        results = tool_runner.run_from_response_content(response.content)
        # 将结果添加到消息列表
        messages.append({"role": "user", "content": results})
```

大致就是这样，[Claude Code](https://code.claude.com/docs/zh-CN/how-claude-code-works) 总结 agent 循环分为三个过程，**收集上下文**、**采取行动**和**验证结果**。这些步骤都需要 tool call，后续要添加大量工具，来完善 agent 的能力。规范工具使用很有必要。

基本功能完成，可以运行后进行测试。基本流程：用户输入问题，调用 LLM 返回推理结果，根据推理结果执行工具或者输出信息，当前一轮结束后，继续等待用户输入，接着开始下一轮的循环。
